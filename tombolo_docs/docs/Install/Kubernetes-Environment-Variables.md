# Kubernetes Environment Variables

This guide explains how to provide environment variables for Tombolo on Kubernetes and how to use Akeyless as your vault backend.

Automatic restarts on Secret/ConfigMap changes are configured through Stakater Reloader annotations in the manifests. Ensure the controller is installed in your cluster.

## Strategy

Use two buckets:

1. ConfigMap for non-secret values
2. Secret for sensitive values

In Helm, `server` and `jobs` consume config using `envFrom`.

## Variable Ownership

Keep these non-secret values in ConfigMap (examples):

- `NODE_ENV`
- `NODE_LOG_LEVEL`
- `SERVER_PORT`
- `WEB_URL`
- `REDIS_HOST`
- `REDIS_PORT`
- `REDIS_DB`
- `JOBS_SERVICE_URL`
- `AKEYLESS_API_URL`
- `AKEYLESS_PATH_PREFIX`

For shared runtime values (recommended), place shared database and Redis connection settings in a central ConfigMap named `tombolo-runtime-config`.

Single-source rule: define `DB_HOSTNAME`, `DB_PORT`, `DB_NAME`, `REDIS_HOST`, `REDIS_PORT`, and `REDIS_DB` only in `tombolo-runtime-config`.

Keep these in Secret:

- `DB_USERNAME`
- `DB_PASSWORD`
- `JWT_SECRET`
- `JWT_REFRESH_SECRET`
- `CSRF_SECRET`
- `ENCRYPTION_KEY`
- `JOBS_API_KEY`
- `REDIS_PASSWORD`
- `AKEYLESS_ACCESS_ID`
- `AKEYLESS_ACCESS_KEY`
- OAuth and SMTP credentials

For shared runtime secrets (recommended), place shared database and Redis auth values in a central Secret named `tombolo-runtime-secrets`.

Single-source rule: define `DB_USERNAME`, `DB_PASSWORD`, `REDIS_USER`, and `REDIS_PASSWORD` only in `tombolo-runtime-secrets`.

## Helm Workflow

Files:

- `Tombolo/helm/tombolo/values.yaml`
- `Tombolo/helm/tombolo/values-dev.yaml`
- `Tombolo/helm/tombolo/values-secrets.local.yaml` (generated by `prepare.sh`)

Recommended pattern:

1. Keep non-secrets in values files.
2. Put secrets in an encrypted values file (SOPS, sealed secrets, or your CI secret store).
3. Pass secret values at deploy time.

Then run the script workflow:

```bash
cp Tombolo/helm/tombolo/values-secrets.example.yaml Tombolo/helm/tombolo/values-secrets.local.yaml
# edit Tombolo/helm/tombolo/values-secrets.local.yaml with secret values

./Tombolo/helm/tombolo/prepare.sh --build-images
./Tombolo/helm/tombolo/run.sh --wait
```

Equivalent manual Helm command:

```bash
helm upgrade --install tombolo Tombolo/helm/tombolo \
  -n tombolo \
  --create-namespace \
  -f Tombolo/helm/tombolo/values.yaml \
  -f Tombolo/helm/tombolo/values-dev.yaml \
  -f Tombolo/helm/tombolo/values-secrets.local.yaml
```

When done, clean up with:

```bash
./Tombolo/helm/tombolo/cleanup.sh
```

## Akeyless (Direct SDK/API Model)

Your current model is direct API access from the app, not Kubernetes CSI/ESO sync. That means pods still need bootstrap credentials and path metadata.

### Required Env Vars

- `AKEYLESS_API_URL`
- `AKEYLESS_PATH_PREFIX`
- `AKEYLESS_ACCESS_ID`
- `AKEYLESS_ACCESS_KEY`

### Runtime Flow

1. Pod starts with Akeyless bootstrap vars from Kubernetes Secret/ConfigMap.
2. Application authenticates to Akeyless.
3. Application fetches secrets from `AKEYLESS_PATH_PREFIX`.
4. If a secret exists in both `.env` and Akeyless, `.env` value wins (existing Tombolo behavior).

### Example Secret Definitions

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: tombolo-server-secrets
  namespace: tombolo
type: Opaque
stringData:
  AKEYLESS_ACCESS_ID: p-xxxxx
  AKEYLESS_ACCESS_KEY: your-access-key
  DB_PASSWORD: your-db-password
  JWT_SECRET: your-jwt-secret
```

### Rotation Guidance

1. Rotate value in Akeyless first.
2. Update Kubernetes bootstrap credential secret if needed.
3. Restart deployments to force refresh if your app loads secrets only at startup:

```bash
kubectl rollout restart deployment/tombolo-server -n tombolo
kubectl rollout restart deployment/tombolo-jobs -n tombolo
```

### Failure Behavior To Plan For

If Akeyless is unavailable, decide and document one policy:

- Fail closed: container exits if mandatory secret fetch fails.
- Fail open with fallback: use local env for non-critical values only.

Fail closed is recommended for production secrets.

## Quick Validation

Check env presence in a running pod:

```bash
kubectl exec -n tombolo deploy/tombolo-server -- env | grep AKEYLESS
```

Check app health after secret wiring:

```bash
kubectl -n tombolo port-forward svc/tombolo-server 3001:3001
curl -f http://localhost:3001/api/status/health
```
